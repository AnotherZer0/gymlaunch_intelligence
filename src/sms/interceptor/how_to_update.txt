SMS Interceptor — How It Works & How to Update
===============================================

HOW IT WORKS
------------
The gymlaunch-sms-interceptor Lambda sits in front of Octopods for all inbound
Twilio traffic. Every SMS received on one of our Twilio numbers, and every delivery
status callback, hits this Lambda first.

For inbound messages (/sms/inbound):
  1. Validates the Twilio HMAC-SHA1 signature — rejects spoofed requests with 403
  2. Identifies which channel the message arrived on (Marketing / Product Updates)
     using the TWILIO_NUMBER_CHANNELS mapping in template.yaml
  3. Searches HubSpot for the sender by mobilephone, then phone (EVERY message).
     If found, stages the last-message properties:
       - last_twilio_sms_message_content = message body
       - last_twilio_sms_received        = today (date picker, midnight UTC)
     If not found: logs and skips the HubSpot update (message still recorded in RDS).
  4. Checks if the body exactly matches an opt-out keyword (case-insensitive):
       STOP, STOPALL, UNSUBSCRIBE, CANCEL, END, QUIT, OPTOUT, REVOKE
       - Removes the channel from sms_subscriptions (multiple-checkbox property)
       - Sets sms_*_opted_out = true
       - Sets sms_marketing_opt_out_date (Marketing channel only)
       - Stages a note for the contact timeline
  5. Checks if the body is an opt-in keyword (START, UNSTOP, YES)
       - Adds the channel back to sms_subscriptions
       - Sets sms_*_opted_out = false
       - Stages a note
  6. Sends ALL staged properties in ONE PATCH. If HubSpot rejects the batch
     (it rejects the whole PATCH when any one property is invalid), retries each
     property individually so the good ones still land, and emails an SES alert
     to ALERT_TO_ADDRESS naming exactly which properties failed and why.
  7. Writes a row to sms_inbound_message in RDS (hubspot_update_status records
     ok / ok_after_fallback / partial: ... / contact_not_found)
  8. Forwards the raw request body to Octopods (so HubSpot Conversations still works)
  9. Returns empty TwiML <Response/> — Twilio handles STOP auto-replies at carrier level

DEBUG DRY-RUN MODE: set DEBUG=1 on the Lambda in the console. Everything runs
(signature check, contact lookup, property staging) but nothing is written — no
HubSpot PATCH, no note, no DB row, no Octopods forward, no email. The HTTP
response becomes a JSON body showing exactly what WOULD happen (would_patch,
would_create_note, contact_id, etc.). Signature failures don't 403 in debug so
you can test with curl. The flag is declared in template.yaml so every deploy
resets it to 0.

For delivery status callbacks (/sms/status):
  1. Validates signature
  2. If error_code is 30006 (geo block), 30007 (carrier violation), or 30008 (carrier error):
       - Finds the recipient contact in HubSpot
       - Sets sms_deliverable = false
       - Sets sms_ineligible_reason to the mapped reason code
  3. Writes a row to sms_delivery_event in RDS
  4. Forwards to Octopods
  5. Returns 200

The Lambda always returns 200 to Twilio, even on internal errors. This prevents
Twilio from retrying the webhook, which would cause duplicate DB rows and duplicate
HubSpot updates.


HubSpot PROPERTIES WRITTEN BY THIS LAMBDA
------------------------------------------
All of these must be created in HubSpot portal 43776308 before deploying.
If a property doesn't exist, HubSpot returns 400 and the Lambda logs it but continues.

Property name                    Type                  Notes
----------------------------     ------------------    ----------------------------
last_twilio_sms_message_content  Multi-line text       Body of the last inbound SMS.
                                                       Written on EVERY message.
last_twilio_sms_received         Date (date picker)    Date of the last inbound SMS.
                                                       Written on EVERY message.
sms_subscriptions                Multiple checkboxes   Enrollment state per channel.
                                                       Internal names MUST match the
                                                       channel names in _CHANNEL_PROPS
                                                       in handler.py (case-sensitive).
                                                       Current options: Marketing,
                                                       Product Updates, Support

sms_marketing_opted_out          Checkbox (boolean)    Set true on STOP, false on START
sms_product_updates_opted_out    Checkbox (boolean)    Set true on STOP, false on START
sms_marketing_opt_out_date       Date (date picker)    Date of most recent Marketing
                                                       STOP. Not cleared on START —
                                                       history is preserved.
                                                       CAREFUL: API name is opt_out,
                                                       NOT opted_out. The wrong name
                                                       shipped originally and silently
                                                       400'd every Marketing STOP.
sms_deliverable                  Checkbox (boolean)    Set false on hard delivery failure
sms_ineligible_reason            Single-line text      Reason code: geo_block |
                                                       carrier_violation | carrier_error

Date-picker properties only accept midnight-UTC epoch milliseconds. A raw
time.time()*1000 value is rejected by HubSpot with INVALID_DATE — use the
_today_midnight_utc_ms() helper in handler.py.


HOW TO ADD A NEW TWILIO NUMBER (same channel)
---------------------------------------------
Example: adding +12135001234 to the Marketing channel.

1. Get the Octopods webhook URL for the new number from Octopods app settings.
   Add it to the gymlaunch/twilio/octopods_webhook_urls secret in Secrets Manager:

     aws secretsmanager get-secret-value \
       --secret-id gymlaunch/twilio/octopods_webhook_urls \
       --query SecretString --output text | base64 -d
     # Note the current JSON, add the new entry, then update:
     aws secretsmanager put-secret-value \
       --secret-id gymlaunch/twilio/octopods_webhook_urls \
       --secret-string '{ ...existing entries..., "+12135001234": "https://app.octopods.io/channels/twilio/hubspot/notification/YOURTOKEN" }'

   The secret value is raw JSON (not base64) — deploy.sh handles the encoding.

2. Open infra/template.yaml.
   Find TWILIO_NUMBER_CHANNELS under SmsInterceptorFunction → Environment → Variables.
   Add the number to the appropriate channel array:
     TWILIO_NUMBER_CHANNELS: '{"Marketing": ["+12135661157", "+12135613526", "+12135795481", "+12135001234"]}'

3. Deploy:
     bash scripts/deploy.sh

4. In the Twilio console: assign the new number to your Messaging Service
   (or set its inbound webhook URL directly to {SmsApiUrl}/sms/inbound, method POST).

If you skip step 1, the Lambda will still process opt-outs and write to RDS, but
won't forward messages to Octopods — HubSpot Conversations will be silent for that number.


HOW TO ADD A NEW CHANNEL
-------------------------
Example: adding a "Product Updates" number +12135009999.

Step 1 — Create HubSpot properties (if not already done):
  - sms_product_updates_opted_out  (Checkbox)
  - Add "Product Updates" as an option to sms_subscriptions if not present
    (the internal name must match exactly what you use in _CHANNEL_PROPS below)

Step 2 — Add channel config in handler.py:
  Find _CHANNEL_PROPS and add an entry:
    "Product Updates": {
        "opted_out":          "sms_product_updates_opted_out",
        "opted_out_date":     None,   # add a date property here if you want one
        "subscription_value": "Product Updates",
    },

Step 3 — Add the number to template.yaml:
  TWILIO_NUMBER_CHANNELS: '{"Marketing": [...], "Product Updates": ["+12135009999"]}'

Step 4 — Deploy:
  bash scripts/deploy.sh

Step 5 — Update Twilio:
  Point the new number's inbound webhook at {SmsApiUrl}/sms/inbound


HOW TWILIO SIGNATURE VALIDATION WORKS
--------------------------------------
Every request Twilio sends to our webhook includes an X-Twilio-Signature header.
We validate it before doing anything else — invalid signatures get a 403 immediately.
This prevents anyone who discovers the public API Gateway URL from posting fake data.

The algorithm (Twilio spec):
  1. Start with the full request URL
     e.g. https://0809f9x39a.execute-api.us-east-1.amazonaws.com/sms/inbound
  2. Sort all POST params alphabetically, concatenate them with no separators:
     key1value1key2value2key3value3...
     Blank-value params (ToCity=, ToZip=) MUST be included — Twilio signs them too.
  3. Sign the concatenated string with HMAC-SHA1 using the Twilio auth token as the key.
  4. Base64-encode the result.
  5. Compare against X-Twilio-Signature in constant time (prevents timing attacks).

We do the same thing when forwarding to Octopods, except we re-compute the signature
using the Octopods URL instead of ours. The signature Twilio sent is bound to our URL —
Octopods validates against their own URL, so we must re-sign before forwarding.

Debugging a signature mismatch:
  - CloudWatch logs print the received sig, computed sig, and param keys when validation fails.
  - Most common cause: URL mismatch (trailing slash, HTTP vs HTTPS, wrong domain).
  - Second most common: a POST param was dropped or double-decoded before validation.
    Make sure you parse the raw body with keep_blank_values=True.
  - If the Twilio auth token was rotated, update gymlaunch/twilio/auth_token in
    Secrets Manager and redeploy — the Lambda reads it from the env var, not Secrets Manager
    at runtime, so a secret rotation alone doesn't take effect until next deploy.


HOW TO FIND THE API GATEWAY URL AFTER DEPLOY
--------------------------------------------
  aws cloudformation describe-stacks \
    --stack-name gymlaunch-slack-sync \
    --query "Stacks[0].Outputs[?OutputKey=='SmsApiUrl'].OutputValue" \
    --output text

Append /sms/inbound for the Twilio inbound webhook.
Append /sms/status for the Twilio StatusCallback URL.


HOW TO CHECK OPT-OUT HISTORY IN RDS
-------------------------------------
-- All opt-outs in the last 30 days by channel
SELECT channel, opt_out_keyword, from_number, received_at
FROM sms_inbound_message
WHERE opt_out_keyword IS NOT NULL
  AND received_at > now() - interval '30 days'
ORDER BY received_at DESC;

-- Opt-out rate by channel (rough — denominator is all inbound, not sends)
SELECT channel,
       COUNT(*) FILTER (WHERE opt_out_keyword IS NOT NULL) AS opt_outs,
       COUNT(*) AS total_inbound,
       ROUND(100.0 * COUNT(*) FILTER (WHERE opt_out_keyword IS NOT NULL) / COUNT(*), 1) AS opt_out_pct
FROM sms_inbound_message
GROUP BY channel;

-- Hard delivery failures by error code
SELECT error_code, COUNT(*) AS count
FROM sms_delivery_event
WHERE error_code IS NOT NULL
GROUP BY error_code
ORDER BY count DESC;

SECRETS USED BY THIS LAMBDA
-----------------------------
gymlaunch/hubspot/token        HubSpot Private App token
gymlaunch/twilio/auth_token    Twilio auth token (for Twilio signature validation)
gymlaunch/twilio/octopods_webhook_urls  Per-number Octopods forwarding URLs (JSON)

The octopods_webhook_urls secret is a JSON object keyed by our Twilio number in E.164:
  {
    "+12135613526": "https://app.octopods.io/channels/twilio/hubspot/notification/KyZWXu-LSNswdGeq",
    "+12135795481": "https://app.octopods.io/channels/twilio/hubspot/notification/FrvtvMO37LYTkjCa",
    "+12135661157": "https://app.octopods.io/channels/twilio/hubspot/notification/U5U3zO49mfKc6aq6"
  }

Each Twilio number has its own Octopods webhook token. If you add a new number,
add its entry to this secret (aws secretsmanager update-secret) then redeploy.

PHASE 3 NOTE: When Octopods is replaced entirely, the OCTOPODS_WEBHOOK_URLS env
var and _OCTOPODS_URL_MAP in handler.py are removed. All inbound traffic routes
through our own Lambda, and all outbound sends go through our own outbound Lambda
with a single StatusCallback URL. The per-number forwarding complexity goes away.
